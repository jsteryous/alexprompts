Depth over volume: about two to three pieces a week set by the schedule, never a daily content mill.

You run in the cloud with a fresh checkout of the repo and zero prior context. Each pass's spec lives in its own file under scripts/greenville/routine/. When a step says to, read that file and hand its FULL contents to the sub-agent for that pass.

PUBLISH MODE: **review** (draft-first). New posts are created as **DRAFT** and do NOT go live on their own. To go back to auto-publish live, change `DRAFT` back to `PUBLISHED` (and `published_at = NULL` back to `now()`) in STEP 3.

WORKSPACE. Do ALL scratch work in /tmp/gv (run: mkdir -p /tmp/gv). Write every intermediate there (evergreen_done.txt, evergreen_topic.txt, pass_draft.md, pass_final.md). NEVER write scratch files into the git working tree and NEVER edit .gitignore. The only repo commands you run are reading the topic bank and the pass spec files.

ISOLATION (the quality lever). STEP 1, PICK THE TOPIC (cadence guard, dedup, bank-first then scout). Run: mkdir -p /tmp/gv. Then, using the Supabase connector (mcp tool):
  1. SAFETY GUARD (the SCHEDULE sets the ~2-3/week cadence now; these two checks only stop an accident). Run both against blog_posts:
     a. SAME-DAY DUPE. `select count(*) from blog_posts where 'evergreen' = any(tags) and created_at::date = current_date;`
        If the count is >= 1, an evergreen piece already went out today (a duplicate run or a retry): STOP, post nothing, report "NO RUN (already ran today)", and end the run cleanly.
     b. DRAFT BACKPRESSURE. `select count(*) from blog_posts where 'evergreen' = any(tags) and status = 'DRAFT';`
        If the count is >= 3, the review queue is backed up (draft-first means unreviewed drafts pile up if Alex gets busy): STOP so the engine never outruns his review, post nothing, report "NO RUN (draft backlog)", and end the run cleanly.
     Both are normal, expected outcomes. If neither trips, continue.
  2. GATING. If the Supabase connector is unavailable or the query fails, you cannot check cadence or dedup. You may still proceed, but you MUST publish as DRAFT in STEP 3 (never auto-publish live without dedup). Note it and continue. (In a full outage the STEP 3 insert will also fail, and you fall back to email-only delivery there.)
  3. TOPIC DEDUP. Pull the slugs AND titles of evergreen pieces already published:
     `select slug, title from blog_posts where 'evergreen' = any(tags);`
     Write them to /tmp/gv/evergreen_done.txt (the scout needs the titles to avoid near-duplicates, not just exact slugs).
  3b. PUBLISHED PAGES (for the writer's internal links). Pull every LIVE local page so the writer can interlink related guides (drafts would 404, so filter on PUBLISHED):
     `select slug, title, tags from blog_posts where status = 'PUBLISHED' and ('greenville' = any(tags) or 'greenville works' = any(tags));`

  4. PICK THE CANDIDATE (bank first, then self-source). Read scripts/greenville/topics.md. Save that to /tmp/gv/evergreen_topic.txt, and record that the topic came from the SCOUT. If the scout reports that NOTHING clears the five bars (rare, because Greenville real-estate long-tail is deep), do NOT proceed: post nothing, report it, and end the run cleanly. Keep the scout's ALSO-RANS for the STEP 4 email.
