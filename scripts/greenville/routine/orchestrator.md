SINGLE TRACK: EVERGREEN LOCAL-SEO. The job of each piece is to win a real, winnable local Google query with buyer or seller intent (relocation, neighborhood, cost-of-living, first-time-buyer, local-investor), stay true for years, and end by pointing leads to /find-an-agent. Depth over volume: about two pieces a week, never a daily content mill.

You run in the cloud with a fresh checkout of the repo and zero prior context. Each pass's spec lives in its own file under scripts/greenville/routine/. When a step says to, read that file and hand its FULL contents to the sub-agent for that pass.

PUBLISH MODE: **review** (draft-first). New posts are created as **DRAFT** and do NOT go live on their own. Alex reviews every piece before it publishes: the STEP 4 email carries the post id and a /review link so he can read it, edit anything that looks off, and publish with one click. The pass guardrails (the evergreen writer's anti-thin-content bar, its strict fair-housing rules, and every number traced to a cited source) and dedup still run, but a human is now the final gate. Nothing is broadcast or covered until Alex publishes: the finalize cron only touches PUBLISHED rows. To go back to auto-publish live, change `DRAFT` back to `PUBLISHED` (and `published_at = NULL` back to `now()`) in STEP 3.

WORKSPACE. Do ALL scratch work in /tmp/gv (run: mkdir -p /tmp/gv). Write every intermediate there (evergreen_done.txt, evergreen_topic.txt, pass_final.md). NEVER write scratch files into the git working tree and NEVER edit .gitignore. The only repo commands you run are reading the topic bank and the pass spec files.

ISOLATION (the quality lever). Run each pass as a separate sub-agent (Task tool, subagent_type "general-purpose") so it starts cold and sees ONLY the input you hand it: its spec file plus the named /tmp/gv input(s). Save each pass output to its /tmp/gv file before starting the next. If you cannot spawn sub-agents, do the passes yourself as clean rooms: finish and save one file before reading anything for the next.

STEP 1, PICK THE TOPIC (cadence guard, dedup, bank-first then scout). Run: mkdir -p /tmp/gv. Then, using the Supabase connector (mcp tool):
  1. CADENCE GUARD (depth over volume, about two pieces a week). Find the most recent evergreen post:
     `select slug, created_at from blog_posts where 'evergreen' = any(tags) order by created_at desc limit 1;`
     If a row exists and its created_at is within the last 3 days, STOP: post nothing tonight, report "NO RUN (cadence)", and end the run cleanly. This is a normal, expected outcome.
  2. GATING. If the Supabase connector is unavailable or the query fails, you cannot check cadence or dedup. You may still proceed, but you MUST publish as DRAFT in STEP 3 (never auto-publish live without dedup). Note it and continue. (In a full outage the STEP 3 insert will also fail, and you fall back to email-only delivery there.)
  3. TOPIC DEDUP. Pull the slugs AND titles of evergreen pieces already published:
     `select slug, title from blog_posts where 'evergreen' = any(tags);`
     Write them to /tmp/gv/evergreen_done.txt (the scout needs the titles to avoid near-duplicates, not just exact slugs).
  4. PICK THE CANDIDATE (bank first, then self-source). Read scripts/greenville/topics.md. Walk the `queued` list top to bottom and select the FIRST topic whose `target_slug` is NOT in /tmp/gv/evergreen_done.txt. If one exists, that is tonight's topic: copy its full entry (title, target_query, target_slug, anchor, and the data-to-ground note) to /tmp/gv/evergreen_topic.txt, and record that the topic came from the BANK. If no queued topic remains (or the bank is empty), run PASS 0, the scout: read scripts/greenville/routine/pass0_scout.md and hand its full contents plus /tmp/gv/evergreen_done.txt (the ALREADY PUBLISHED titles + slugs) to a fresh sub-agent. Save that to /tmp/gv/evergreen_topic.txt, and record that the topic came from the SCOUT. If the scout reports that NOTHING clears the five bars (rare, because Greenville real-estate long-tail is deep), do NOT proceed: post nothing, report it, and end the run cleanly. Keep the scout's ALSO-RANS for the STEP 4 email.

STEP 2, WRITE IT (pass_evergreen). Read scripts/greenville/routine/pass_evergreen.md. Hand its full contents plus /tmp/gv/evergreen_topic.txt to a fresh sub-agent (subagent_type "general-purpose").
